# 来源核验(差异化护城河)

这是本 Skill 区别于普通搜索 / Coze 的核心。每份简报都要让用户**一眼看清信息从哪来、可信几分**,但不要把内部风控标签直接堆到前台。

## 四分区
把进入简报的信息按来源归四类,最终汇总到 `references[]` 和 `source_check`:

| 分区 | 含义 | 处理 |
|---|---|---|
| 官方 official | 公告、交易所、监管、央行、公司公开信息 | 可直接采信 |
| 媒体转述 media | 主流财经媒体报道 | 关键事实需 ≥2 源印证 |
| 机构观点 institution | 券商研报/机构判断 | 订阅内容标「待原文核验」;区分事实与观点 |
| 内部剔除 internal_excluded | 二手转述、传闻、社区小作文 | 默认不进入前台;仅用于审计 |

## 核心事实的一手来源优先级

宏观、政策和跨市场问题里,核心证据不能主要靠低可信转引站点承担。先找一手或准一手来源,再用媒体/机构解释市场反应:

- 央行与利率:Fed、FOMC、ECB、PBOC、CME FedWatch、交易所或清算所官方页面。
- 通胀与就业:BLS、BEA、统计局、财政部、央行数据库。
- 能源与大宗:EIA、IEA、OPEC、交易所/期货交易所官方数据。
- 债务与财政:美国财政部、CBO、国会/财政部门官方发布。
- A 股盘面:交易所、指数公司、权威行情源、主流财经媒体的盘面复盘。

写 `references[]` 时,前 5 条应优先放能支撑核心事实、影响链和验证信号的一手/权威来源;其余背景来源可以保留,但前台会默认折叠。媒体转述可以用于「市场反应」「机构观点」「背景解释」,不要让它替代官方数据或央行公告。

### 央行与利率问题

`policy_macro` 的央行与利率问题要把事实层和市场定价层分开:

- 当前任职人、会议日期、利率区间、政策声明、会议纪要和预测材料必须优先引用央行官网、交易所/清算所或官方数据页;中文媒体只作市场反应或背景说明。
- 市场概率必须带 as_of:例如 FedWatch、期货隐含概率、预测市场概率都要写明抓取日期/时间、平台名称和口径,不能把动态概率写成静态事实。
- 动态概率不得替代央行官方事实;正文应说明这是「市场定价」或「隐含概率」,不是央行承诺。
- 若缺少一手/准一手来源,`references[]` 不得只列来源名称;应记录 `coverage_gaps[]`,并降低核心结论信心。

## 多源印证度
- 同一事实被几个**独立**来源印证 → 记 `corroboration`(整数),前台写「多源印证」。
- 仅单一来源 → `corroboration:1`,前台弱提示「单一来源」,不要抢占主信息。
- 印证源必须独立(互相转载不算多源)。

## 前台来源露出
- 模板只在标题栏展示 `meta.source_scope` 的信源范围说明,不在首屏额外渲染「主要来源」列表。
- 底部信息来源默认展示最多 5 条核心来源;超过 5 条时其余来源折叠,且不展示“默认展示X条/其余X条折叠”这类摘要句。所以 references 的排序必须把最重要、最权威、最能支撑正文的来源放在前面。
- `source` 必须写用户看得懂的名称,如「上交所」「证券时报」「财联社」「中信证券」,不要写泛化的「媒体」「机构」。
- `url` 必须是真实 http(s) 地址;没有可靠原文链接时,这条来源不应承担核心事实。
- `references[].source/url/date/corroboration` 会进入新闻卡、观点原文链接和引用 `[n]` popover;每条解读必须回指至少一个公开来源。

## 来源身份去重
- **同一网页只能有一个 `references[].id`**。即使同一篇文章支撑多个事实、观点、影响链或验证信号,也复用同一个 id,不要按事实拆成多个来源 item。
- 判断同一网页时按规范化 URL:忽略 `utm_*`、`spm`、`from`、fragment、大小写、`www.`、默认端口和末尾斜杠;保留 `page`、`id` 等会改变正文内容的查询参数。
- 如果同一事实有多个独立网页印证,保留多个独立 `references[]`,并把主引用的 `corroboration` 写成独立来源数量;不要把同一网页的多个段落算成多源印证。
- `validate_briefing.py --strict` 会拦截重复 canonical URL。报错时应合并引用:保留更准确的标题、来源、日期和 URL,然后把 `news_items[].ref`、`interpretation[].refs`、`visuals.impact_facts[].refs`、`verification.*[].refs`、`source_check` 中的原 id 全部改到保留 id。

## 引用账本

最终 HTML 的「信息来源」是紧凑来源账本。正文中的 `[n]` 引用负责把关键事实、观点、影响链和验证信号回指到来源;来源区列编号、来源、日期、标题和原文链接,不再逐条展开冗长的「支撑:...」说明。

每条公开来源仍应尽量能被回指到至少一个前台对象:

- `news_items[].ref`:支撑关键增量事实。
- `visuals.impact_facts[].refs`:支撑对应影响链步骤节点正文的来源。
- `interpretation[].refs`:支撑各方观点或反向证据。
- `verification.*[].refs`:支撑验证或证伪信号的观察口径。

渲染时保留轻量来源摘要,正文引用编号与来源列表一一对应。

## 证据原子
前台不是证据库,但后台必须可追溯。每条重要事实先整理成 `evidence_atoms[]`:

```
{
  "claim": "一句可验证事实",
  "source": "来源名",
  "url": "https://...",
  "date": "发布日期或原文时间",
  "excerpt": "原文关键摘录",
  "context": "口径、范围、是否转述",
  "confidence": "HIGH|MEDIUM|LOW",
  "refs": [1]
}
```

- `HIGH`: 官方/监管/交易所,或两个独立权威源一致。
- `MEDIUM`: 单一主流媒体/机构来源,或多源但口径仍需确认。
- `LOW`: 社区、传闻、单一匿名说法;默认不进前台,只进入 `internal_excluded` 或 `coverage_gaps`。
- 同一事实存在冲突时写入 `conflicts[]`,并在洞见中用「分歧集中在…」表达,不要选择性忽略。

## 图片来源
- 关键增量信息不渲染新闻配图,不要给 `news_items[].image` 填任何图片。
- 官方图、原文配图、产品路线图等如果包含关键事实,用文字摘要写入 `news_items[].why`、`visuals.impact_facts` 或 `references[]`;不要把图片作为前台证据入口。
- 社区图、无法确认来源的截图、AI 生成的新闻现场图不进入前台。

## 红线
- 媒体转述**不能替代**公告;涉及确定性事实以官方为准。
- 二手转述、传闻、社区小作文默认丢弃。
- 若因官方回应或重大多源报道必须纳入,前台写成「争议点」或「官方回应」,不得出现「待核验」字样。

## 在 schema 中的落点
来源信息收敛为单一 `references[]` 对象(含 `level` A/B/C/D、`type` official/media/institution/background/unverified、`corroboration`、`date`、`url`);正文与 `news_items` 只引 `id`,不重复表达来源等级,避免三处打架。`type=unverified` 或 `level=D` 仅作内部审计,默认不渲染为新闻卡。`query_log`、`evidence_atoms`、`coverage_gaps`、`conflicts` 是内部审计字段,`render.py` 注入 HTML 前会移除。
