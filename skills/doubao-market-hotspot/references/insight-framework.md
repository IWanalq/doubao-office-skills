# 洞见框架

目标:避免简报沦为「信息罗列」。每份简报必须把公开信息转成一个可验证的市场解读,但不提供买卖建议。

## 必答 7 问
1. **一句话判断**:这批信息合起来说明了什么?
2. **为什么现在重要**:是新政策、新订单、新预期修正,还是过时信息重炒?
3. **真正变化**:相比上一阶段,事实、预期或节奏哪里变了?
4. **各方观点**:客观事实、市场反应、机构观点、反向证据分别在说什么?哪些是事实,哪些是判断?
5. **正向条件**(`positive_case`):有哪些事实会让核心观点更站得住?
6. **反向条件**(`negative_case`):有哪些事实会削弱或推翻核心观点?
7. **验证 / 证伪信号**(`verification.confirm` / `verification.invalidate`):后续看哪些事实来确认或推翻主线?

## 前台 7 段叙事链

HTML 必须按固定链条阅读:

1. 核心观点:只给一句核心判断和 3-4 个关键证据,不展开完整推理;前台不显示 1/2/3 编号,用关键数字和强调色做视觉锚点。
2. 关键增量信息:只写新增事实、时间、来源和一段事实摘要;前台不出现「入选原因」标签,也不要把来源名写进摘要正文。
3. 为什么重要:只写相对预期或上一阶段的真正变化。
4. 如何影响市场:只写因果传导,并把 `visuals.impact_facts[].step` 挂到对应传导节点,让每一步标题下都有可回溯的一段节点正文。
5. 各方观点:按来源角色分组组织客观事实、市场反应、机构观点、反向证据;同一角色只显示一次,组内可放多条观点,每条观点要像小摘要,不是目录。
6. 验证/证伪信号:写成信号名 + 两句完整叙述,不要使用「看什么 / 说明什么」字段标签。
7. 信息来源:列出来源编号、来源、日期、标题和原文链接;正文中的 `[n]` 负责回指支撑关系。首屏不再额外渲染「主要来源」脚注行;公开来源超过 5 条时,底部默认只展示核心来源,其余折叠,且不展示来源数量摘要句。

`news_items` 中不要写「这说明」「意味着」「支撑主线」等解释性话术;这些应进入 `insight`、`impact_facts` 或 `interpretation`。也不要写「某某来源公开信息显示」「据报道」「发布时间为」,来源、日期和链接会由模板展示,正文只写事实本身。

## 信息颗粒度硬规则

- `news_items[].title` 负责一句话说发生了什么;`news_items[].why` 负责补充标题没说清的依据、影响范围和限制条件。不要把标题换个说法再写一遍。
- `news_items[].why` 优先写数字、对象、范围、市场反应或背景。例如「有色金属和券商成为沪指站上4000点的主要拉动力,两市成交继续放大,显示增量资金仍在场内寻找低位方向」,不要写「上海证券报的公开信息显示,低位资源和券商爆发」。
- `visuals.impact_chain` 写短链路,`visuals.impact_facts` 写每个节点标题下的一段正文。每条 `impact_facts` 尽量用 `{step,fact,refs}` 指向对应步骤,`fact` 要让用户直接理解「这一步为什么成立、通过什么机制传导、对市场定价意味着什么、还需要什么事实验证」。
- `interpretation[].point` 要同时回答「谁怎么看」「为什么这么看」「分歧在哪里」,不能只写「机构认为后续仍需观察」。
- `section_summaries.*` 是历史字段名,前台显示为「关键洞见」。它不是本节摘要,必须写二阶判断:真正分歧、预期差、传导断点、最小验证信号或迫使判断下修的条件。不要写「重点关注」「并列跟踪」「官方强调、机构认为」这类罗列式 summary。
- 大标题下面的文字要像论文摘要的浓缩版:两三行内把核心信息讲完。读者不点任何原文链接,也应该能解释这个问题的大概来龙去脉。

## 写成人话(可读性,重要)

`insight` 字段给「这些信息为什么重要？」提供变化和预期差素材,要像一个懂行的人在口头解释,不是研报摘要,也不是字段标签。

- 每个字段写成**连贯、完整的句子**,主谓宾齐全,可以适当写详细一点;用「因为 / 但 / 目前来看 / 接下来 / 关键在于」等连接词把因果和转折讲顺。
- 不要写半截话、不要堆术语、不要把一句话压成「节奏验证」「预期转谨慎」这种电报体。判断要给得明确,但语气是解释而不是下命令。
- `one_sentence` 是例外:它同时当大标题,保持精炼有力。其余字段(why_now / what_changed / consensus_vs_delta / positive_case / negative_case)都可以展开成自然的一句话。
- 写法对比:
  - ✗ 偏术语、太精简:「焦点切到出货验证。预期转谨慎。」
  - ✓ 连贯、像人话:「市场关注点已经从『长期有没有需求』转向『近端能不能放量』,因为海外路线更新叠加机构下修,大家开始重新算兑现时间。」

## 分析模式写法

- `market_event`: 洞见必须覆盖「事件是什么」「相比预期变化在哪里」「接下来什么事实会确认或推翻」,分别落到核心观点、变化和验证/证伪信号里。
- `policy_macro`: 必须把政策/宏观冲击翻译成变量,再翻译成行业或市场环节。
- `dated_catalyst`: 必须写清日期是确定日期还是观察窗口,不要把窗口写成确定事件。
- `earnings_event`: 仅在财报季或业绩风险已经上升为市场级事件时使用,写市场预期、风格和风险偏好的变化;若用户要单家公司财报/业绩数字,转「财报与业绩分析」。
- `theme_watch`: 仅在主题被用户明确作为市场交易主线或风险偏好线索时使用,写主线证据和验证路径;若用户要行业长期观点或板块热度,转「行业分析」或「板块热度分析」。

## 各方观点写法
`interpretation[]` 不是 AI 观点列表,而是把各方(客观事实/市场反应/机构观点/反向证据,必要时补充媒体报道)在说什么摆在一起、带出处、可保留分歧:
- 每条 `point` 先给角度,再给论据,最后能回指 `refs`。
- 每条必须有 `refs:[id]`,优先混合官方/媒体/机构来源,让读者看出信息从哪来;这里的混合来源指独立网页,同一网页支撑多个观点时复用同一个 `references[].id`,不要拆成多个来源。
- 不要为观点另写 `title` 或 `headline`;HTML 直接显示 `point` 正文,引用 `[n]` 接在正文句末。
- 写法示例:「官方路线更新确认长期需求,但机构报告把分歧拉回近端出货节奏。」
- 不要写「值得关注」「持续跟踪」这种无来源、无判断的空话。

## 允许与禁止
- 允许:讲市场方向、受影响环节、节奏、预期差、验证路径。
- 禁止:目标价、买卖点、仓位、涨跌承诺、受益股名单、公司基本面结论、行业长期投资价值判断、财报数字深度解读。
- 若证据不足,必须写「信心低」和缺口,不要硬做判断。

## 反信息罗列检查
如果 `insight.one_sentence` 去掉后,页面仍像新闻列表,说明洞见不合格。
如果 `positive_case` 与 `negative_case` 都能同时成立,必须给出 `verification.confirm` 和 `verification.invalidate`。
如果 `interpretation[]` 去掉 `refs` 后仍然成立,说明它太像泛泛分析,需要回到证据重写。
