# 对话框输出契约

本文件是对话框输出的唯一 Markdown 契约。目标是让聊天窗口里的正文和 HTML 展开版使用同一条叙事链、同一组栏目和同一批事实字段,避免模型自由改成政策报告、研报、监测表或问答式目录。

默认输出是对话框里的深度 Markdown 正文,不是 HTML 的短摘要、不是文件入口、不是检索过程、也不是表格截图。用户不打开 HTML,也应能读懂结论、证据、传导、分歧、验证信号和来源边界。

## 硬性原则

- **顶层栏目必须和 HTML 一样**:只允许使用下面 7 个 HTML 栏目标题,顺序不能变,标题文字不能改写。
- **不得新增同级标题**:不要写「政策联动与传导路径」「各方观点与市场分歧」「信息来源与边界说明」「一、政策背景」等自拟标题。
- **不得输出过程**:正文只保留用户可读结论、证据、分歧、验证和来源,不写执行计划、检索状态或文件操作记录。
- **不得用表格替代叙事**:确认/证伪、来源角色、影响链都用分组 bullet,不用 Markdown 表格。
- **来源必须可点击**:信息来源区必须保留原文 URL;正文关键事实尽量贴 `[1]` 这类引用编号。

## 固定阅读顺序

1. **核心观点**:先写「核心判断: ...」一句话直接回答用户问题,再写 2-4 句摘要,最后用「为什么这么看:」列 4-6 条关键依据。这里必须讲清“发生了什么、市场为何反应、判断的限制条件、主要分歧”,不能只报指数或结论。
2. **有哪些关键增量信息？**:列 2-4 条新增事实。每条都要包含新增事实、数据/口径、相对上一阶段或市场预期的变化、来源编号;不要写「整理」「确认日期」「表格」等过程词。
3. **这些信息为什么重要？**:至少写 3 条,分别回答「现在重要」「真正变化」「共识与分歧」;复杂问题再补「支持主线的条件」「削弱主线的条件」。不要只写“利好/利空”,要解释市场正在重新定价的变量。
4. **上述信息如何影响市场？**:先写一段总结,再按「事实 → 变量 → 资产/市场反应 → 后续验证」拆成 3-5 条。每条必须有机制解释,不能只写箭头链。
5. **各方有哪些值得关注的观点？**:先写一段总结,再至少写 3 条,把官方事实、市场定价、机构观点、反向条件区分开。不要只写“多方/空方”;要说明每方观点依赖什么事实、哪里存在分歧。
6. **哪些信号会验证核心观点？哪些信号代表观点证伪？**:先写一段总结,再分为「确认主线」和「削弱或推翻」。每组 2-3 条可观察信号,每条都要写观察口径和触发含义。
7. **信息来源**:列核心来源,包括来源名、日期、标题和原文链接。来源行要让读者知道这条来源支撑什么,不能只写来源名称。

7 个栏目之后再写:
- **边界**:说明这是公开信息梳理和验证条件拆解,不构成投资建议。
- **HTML 选项**:最后一行必须逐字输出「需要为你生成一份精美的热点报告么？」,不得改写、润色或补写「HTML 格式」「解读简报」等词;只有已生成 HTML 时才给「详细 HTML 简报: <链接>」。

## 写法规则

- 默认不用 Markdown 表格;手机窄屏会变丑,也容易把信息压成字段。强弱板块、来源角色、确认/证伪都用分组 bullet。
- 每个 bullet 先给短标签,再给解释。例如「**市场定价**: ...」「**反向证据**: ...」。
- 来源编号紧贴事实或判断,不要只在最后堆媒体名。
- 深度正文以讲清固定叙事链为准,不要为了铺陈而扩写,也不要因为过度删减丢掉必要证据和反向条件。
- 每个非来源栏目至少给一段解释或 2 条以上要点。不要只用一句话结束一个栏目。
- 专业词第一次出现要顺手解释。能说「市场担心钱会流走」,不要说「流动性约束边际强化」。
- 不输出工具过程、检索过程、文件编辑、内部推理或生成计划。
- 不写目标价、买卖点、仓位、受益股名单、涨跌承诺。
- 不写自拟编号报告目录,例如「一、政策背景」「二、当前立场」「三、市场共识」「四、监测表」「五、核心结论」。如果需要分层,只能使用上面的固定标题。
