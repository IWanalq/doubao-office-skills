## 适用场景

当用户输入股票名称、股票代码、证券简称，或提出任何围绕单只股票的分析问题时，使用本 skill。问题可以是标准每日复盘，也可以是更个性化的目标价、短线交易、事件催化、持仓处理、估值贵不贵、风险是否变大等。

典型触发问题：

- “帮我分析下某某公司 / 生成某股每日复盘 / 做一份个股分析 HTML。”
- “我知道英伟达是好公司，但股价还能到什么位置？”
- “我喜欢短线操作，中船气体现在能不能入手快赚一笔？”
- “中通客车 5 月销量增长，这对公司股价会有拉升吗？”
- “我已经持仓/空仓，今天该拿、减、等还是做短线？”
- “这个公告、订单、业绩、行业新闻对公司股价影响大不大？”

目标是端到端完成：

1. 识别股票和用户真正关心的问题。
2. 获取行情、资金、基本面、消息、技术和预期数据。
3. 形成机构风格、但能直击用户问题的个股分析。
4. 默认先输出对话版完整日报，并询问用户是否继续生成 HTML。
5. 用户确认后，再生成一个自包含、可本地打开、可分享的单文件 HTML。
6. 页面视觉、结构、指标卡、表格和章节顺序必须稳定复现；当前版本不生成 K线模块。

## 规范加载

本 skill 是标准多文件目录。执行时先读取目录根部 `SKILL.md`，再按文件导航在需要时读取 `references/` 与 `scripts/` 中的具体文件。不要把规则自由改写成另一套版式。

## 总工作流

### 1. 股票识别与用户问题识别

先同时识别两件事：

- 股票主体：股票名称、证券代码、市场、币种。
- 用户问题：每日复盘、目标价/上涨空间、短线交易、事件催化、持仓处理、风险判断、估值贵不贵等。

必须把用户问题转化成 `question_focus`，并贯穿对话版日报和后续 HTML 正文。不要只生成模板化日报而不回答用户原问题。

常见 `question_focus`：

- `target_price`：用户关心股价能到哪里。重点看技术压力位、震荡区间、投行一致预期、估值分位、业绩弹性，给出情景区间而非确定目标价。
- `short_trade`：用户想短线快进快出。重点看今日量价、资金流、可核验技术指标、支撑压力和波动风险；只能给研究视角的短线环境判断，不给具体买卖、仓位、止损或止盈指令。
- `event_catalyst`：用户问销量、订单、公告、政策、行业新闻是否会拉动股价。重点核查事件真实性、收入/利润贡献、持续性、是否已被股价反映，并说明对短期情绪和中期基本面的不同影响。
- `position_advice`：用户提到持仓、空仓、被套、盈利。重点分持仓状态给出操作边界和风控。
- `daily_recap`：用户只要求复盘。按标准日报生成，但仍要提炼最重要矛盾。

如果用户问题不清楚，先用最接近的 `question_focus` 推断；只有股票主体无法确定时才向用户确认。

- A股：支持 `300308`、`SZ300308`、`300308.SZ`、`中际旭创` 等。
- 港股：支持 `00700`、`0700.HK`、`腾讯控股` 等。
- 美股：支持 `NVDA`、`MSFT` 等。

如果名称可能对应多只股票，先用搜索或行情接口 disambiguate；无法确定时向用户确认。

### 2. 数据获取

必须获取或尽力获取以下数据。近期、实时、行情、新闻、预测都属于高时效内容，必须联网或调用可用行情/数据工具，不要只凭记忆。

数据验收先行硬规则：

- 本报告的数据准确性高于生成速度和页面完整度。任何关键数据只要日期、市场、粒度、单位或来源有误，后续分析都必须视为无效。
- 在写任何分析段落前，先确定 `report_request_time`（用户发起请求时间）、`data_as_of`（行情截至时间）和最近有效交易日。若请求发生在周末、节假日、休市后或停牌期间，应按交易所日历向前推到最近一个有真实交易的交易日；如果当日盘中请求，则以请求时点可获得的最新行情为准。
- 先纵览报告需要的全部数据字段，再统一取数和核验。不要边写报告边临时补数据，除非发现必要缺口；补充数据也必须先核验。
- 必须先形成内部数据核验记录。该记录不要求保存为本地 `.txt`，也不得作为报告信息来源。记录至少包括：股票身份、请求时间、最近有效交易日、数据截至时间、每个字段的值、单位、来源 1/2/3、可打开原始外部链接、核验结果、缺失原因、自行计算公式、新闻/公告原文链接、最终是否允许进入报告。
- 后续对话版日报和 HTML 报告原则上只能使用已核验数据。内部记录中标记为“未核验/冲突/可疑/缺失”的数据，不得进入正文判断、表格或技术分析。
- 每个重要数据尽量用 2-3 个独立渠道核对；至少要保证同一字段的证券代码、交易市场、交易日期、时间点、复权口径和单位一致。若来源冲突且无法判断，以“缺失”处理，不要平均、猜测或挑一个顺眼的数。
- 经过至少 3 次换源、扩大请求区间、修正代码/市场、修正复权或分页后仍拿不到的字段，可以留空；不得为了完整而伪造或降格放行。

取数入口硬规则：

- 在豆包等支持专用搜索工具的环境中，必须先调用 `seed_finance_search` 与 `general_search` 搜集和交叉验证数据。
- `seed_finance_search` 优先用于证券识别、实时/收盘行情、量比、PE TTM、资金流、板块、技术指标、一致预期等金融结构化数据。
- `general_search` 优先用于公司公告、交易所/监管披露、权威财经新闻、产业媒体、行业事件和信息来源原文。
- 只有当上述两个工具不可用、字段不足、数据校验失败或覆盖市场不匹配时，才继续使用同花顺、华泰证券、老虎证券等 MCP；再失败时使用腾讯、AkShare、Tushare、东方财富、Yahoo/yfinance、Google Finance、Stooq、交易所或其他稳定源。
- 不论来源优先级如何，都必须先通过数据校验，再进入正文分析；优先源返回不完整数据时不能降格放行。

数据来源前置标准化硬规则：

- 当前版本不获取 1 分钟级行情，不获取 30 个交易日日线，不获取 90 个交易日行情，不生成 `window.RECAP_CHART_DATA`，不生成 K线图。
- 所有进入报告的数值必须先进入内部数据核验记录，并记录为统一结构：`字段名 / 数值 / 单位 / 时间口径 / 来源名称 / 原始外部来源链接 / 是否核验 / 是否进入报告`。
- 行情摘要、资金流、板块行情、PE TTM、量比、MA、MACD、RSI、投行一致预期等数据必须优先使用数据源直接披露值。不同平台对技术指标和量比的算法可能不同，不再要求 agent 自行重算作为主口径。
- 若数据由 agent 自行计算，只能作为兜底，并必须写入内部数据核验记录：输入字段、输入来源链接、公式、计算过程、计算结果和“自行计算”标记。HTML 或对话版中引用时仍要给原始输入来源角标。
- 凡找不到可打开来源链接、无法核验、日期/市场/单位冲突或口径不明的数据，不得进入正文、表格、评级或风险判断。

必须获取：

- 摘要指标：当前价格/收盘价、涨跌幅、日内区间、总市值、市盈率 TTM、成交额、换手率、量比。
- A 股资金流：主力资金净流入/流出、北向资金、大单资金或大单成交占比。非 A 股不生成 A 股资金流表，只获取可核验交易结构线索，如成交量相对均量、南向资金、期权异动、ETF 流向、机构持仓变化、盘前盘后表现等。
- 相关板块：最相关的 1-3 个行业/概念/主题板块，当日涨跌、涨跌家数、龙头/权重地位。
- 技术指标：MA、MACD、RSI，以及震荡区间、压力位/压力区间、支撑位/支撑区间。
- 消息：最近 3 日内影响公司业务和股价的权威财经新闻、产业新闻、公司公告、交易所/监管披露、行业事件；未来当天剩余时间和未来几日的潜在催化。
- 投行一致预期：A 股或有完整预测时获取未来 3 年归母净利润、营业收入、EPS、PE；港股、美股或主要披露券商评级/目标价时，获取最近约 5 家机构的机构名称、最新评级、目标价、日期。
- 风险：当前走势风险、估值风险、业绩/业务风险、流动性和情绪风险。

按 `question_focus` 补充取数：

- `target_price`：补充一致预期 EPS/PE、历史估值区间、可核验压力位/支撑位、上方缺口/前高/整数位、可比公司估值。
- `short_trade`：补充当日收盘/盘中快照、成交额相对 5 日均值、短线资金流和可核验关键波动区间，用于说明风险环境，不用于生成具体交易指令。
- `event_catalyst`：补充事件公告/新闻原文、事件发生日期、市场是否已交易该事件、对收入/利润/订单/产能/政策预期的可能影响。
- `position_advice`：补充成本区间无法获知时，用当前价上下关键位给出通用持仓/空仓/短线三类建议。

详见本文后部“数据字段与计算口径”。

技术指标核验：

- MA、MACD、RSI、量比、换手率等技术/交易指标必须优先取自可打开、可核验的数据源直接披露值，并在内部数据核验记录中记录来源链接。
- 由于不同平台算法和复权口径不同，技术指标必须至少用 2 个来源核对方向和关键读数；若读数不完全一致但方向一致，可以展示主来源读数并在内部数据核验记录中注明差异。
- 若来源冲突无法解释，不展示具体读数，只保留审慎文字判断或写“技术指标暂未核验”。
- 自行计算只可作为兜底；必须披露公式、输入数据来源、参数和“自行计算”标记，不得把自行计算结果伪装成平台披露。

新闻、公告、研报与一致预期核验：

- 新闻和公告必须核查最近 3 日内的原文、发布日期、来源机构和是否与该公司/行业直接相关；过早、无原文、无权威来源或只讨论资金异动的内容，不得作为第四部分“消息面复盘”的主依据。
- 研究报告和一致预期先查 `seed_finance_search` 与 `general_search`，再查 Wind、FactSet、Choice、东方财富、券商研报或公司/交易所披露。若只取得 EPS、目标价或评级等部分字段，必须说明表格口径和自行计算项；表格下方保留简短口径说明。

### 3. 分析写作

报告要像交易台盘后快报，不要像泛泛新闻摘要。

写作要求：

- “今日行情、量能与资金流向”是开头核心段，必须把价格走势和成交量、换手率、量比、资金流结合解释，并进一步回答“为什么会这样走”。价格和量能是结果，必须尝试追溯到公司基本面、行业景气、客户/产品/订单、政策/宏观、海外映射、板块风险偏好或重要公告等原因。若没有检索到明确原因，必须写明“未见明确基本面催化，盘面更偏交易结构/板块情绪驱动”，不得臆造。
- 若用户提出个性化问题，第一部分必须明确说明今日盘面是否支持用户的判断：例如目标价问题要说明当前量价是否支持继续上攻；短线问题要说明追高、回踩或突破确认哪个更合适；事件催化问题要说明盘面是否已经反映该事件。
- “板块联动与个股地位”要说明个股相对板块是否强势、是否带动板块、是否只是跟随 beta。
- “技术面分析”要把 MA、MACD、RSI 分别拆成：短标题、具体内容、状态标签；关键价位要落到区间带图和表格。具体内容必须照顾不懂技术指标的读者，用 1-2 句完整中文解释“这个指标现在说明什么”，不要只堆 MA5/MA10/DIF/DEA/RSI 数字。
- “消息面与隔夜观察”必须引用最近 3 日内的权威财经媒体、产业媒体、公司公告/IR、交易所/监管披露等来源，聚焦公司基本面、产业基本面、行业趋势和未来催化；不要把成交额、大宗交易、龙虎榜、主力净流入等资金异动新闻重复写成消息面。
- “综合评估”要给出研究评估和充分评估理由，必须包含基本面或行业趋势判断，并结合估值、技术、资金或消息面证据；不得输出空仓者、持仓者、短线参与者等具体操作建议表格。
- 第六部分必须直接回答 `question_focus`：目标价问题给情景区间与失效条件；短线问题只说明短线交易环境、波动和风险，不给具体买卖/仓位/止损/止盈指令；事件催化问题给短期情绪影响与中期基本面影响；持仓问题只能给研究视角的风险边界，不给个性化交易指令。
- “短期风险提示”要对散户友好，重点提醒追涨杀跌、估值高位、消息不及预期、客户/行业/资金面变化。

### 4. 对话版完整日报输出

由于完整 HTML 生成和页面复核耗时较长，调用本 skill 时默认先输出“对话版完整日报”，不直接生成 HTML。对话版日报必须承载完整观点和分析深度，不能只给 3-5 条简短摘要。

触发时机：

- 完成核心数据抓取、校验和分析后立即输出。
- 核心数据至少包括：当前价/涨跌幅、成交额/换手率/量比、A 股主要资金流或非 A 股可核验交易结构线索、最相关板块表现、关键技术位、最近 3 日基本面/产业面消息或“未检索到权威消息”、一致预期或缺失说明。
- 若个别数据缺失，必须在正文中标注缺失项、替代来源或估算口径，不得编造。

内容要求：

- 对话版正文最开头必须先输出固定提示语：`「以下内容仅为行情与逻辑分析，不构成任何投资买卖建议，务必结合自身风险承受能力决策」`；该提示语位于报告标题之前，且只出现一次。
- 标题固定为 `个股日报 / 对话版深度复盘`。
- 开头先写 `核心结论`，列 3-5 点，第一点直接回答用户问题。
- 正文必须按以下固定结构输出，不得只给摘要：
  1. `一、今日行情、量能与资金流向`
  2. `二、板块联动与个股地位`
  3. `三、技术面分析`
  4. `四、消息面与隔夜观察`
  5. `五、投行一致预期`
  6. `六、综合评估`
  7. `七、短期风险提示`
  8. `信息来源`
  9. `免责声明`
- 每个主体章节至少 1 段深度分析；第一、三、四、六部分应更充分，避免空泛。
- 不要把 8 个摘要指标逐项重复一遍；必须把数据转化为判断。
- 不要使用“必涨、稳赚、满仓”等绝对化语言。
- 不得以报告自身口吻输出直接投资操作建议，例如“建议买入 / 卖出 / 加仓 / 减仓”“适合买入”“应减仓”等。允许在客观转述、条件假设、风险提示或用户既有状态描述中出现相关词汇，但必须明确保持研究分析口径，不得形成操作指令或替用户作出交易决策。
- 正负颜色语义仍按 A 股习惯：涨/净流入/正数为红，跌/净流出/负数为绿。
- 若用户问“股价能到什么位置”，核心结论第一条必须写成情景判断：上方第一观察区间、乐观情景触发条件、失效条件，不能给确定承诺。
- 若用户问“短线能不能快赚”，核心结论第一条必须从研究合规角度说明当前短线波动环境和风险，不得给具体买卖、仓位、止损或止盈指令。
- 若用户问“某事件会不会拉股价”，核心结论第一条必须先回答“短期情绪催化/中期基本面支撑/影响有限/已被定价”，并写明事实依据。
- 结尾必须询问用户是否需要继续生成完整 HTML 报告，不要自动开始生成 HTML。

对话版投行报告格式硬规则：

- 输出前必须先读完本格式规则，再开始写正文；不得每次临时换一套“投行风格”。
- 使用 Markdown 表达固定层级，不输出 HTML 卡片、不使用 emoji、不使用彩色背景、不使用夸张营销标题、不使用多余分割线。
- 对话版最顶部固定输出 `「以下内容仅为行情与逻辑分析，不构成任何投资买卖建议，务必结合自身风险承受能力决策」`，视觉等效 13px、常规灰色；该提示语必须位于主标题之前，且全文只出现一次。
- 字体口径统一为系统中文无衬线：`-apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", Arial, sans-serif`。对话框无法控制真实字体时，也必须按此口径组织字号、粗细和层级。
- 报告主标题固定为 `**个股日报 / 对话版深度复盘**`，视觉等效 18px、700、行高 1.35；标题下方一行元信息写 `股票简称 证券代码 · 生成日期 · 研究讨论，不构成投资建议`，视觉等效 12px、常规灰色。
- 一级章节标题固定使用中文编号和加粗，如 `**一、今日行情、量能与资金流向**`，视觉等效 16px、700、段前 18px、段后 8px。
- 小标题用行内加粗冒号，如 `**价格与量能：**`、`**资金流向：**`，视觉等效 14px、700；不要另造三级标题样式。
- 正文视觉等效 14px、行高 1.72；每段 80-160 个汉字为宜，段间空一行；不要把整段写成过长新闻稿。
- `核心结论` 用 3-5 条编号列表，每条开头必须是加粗主题词，例如 `1. **直接回答：** ...`；第一条直接回应用户问题。
- A 股资金流、技术指标、投行一致预期等需要表格时，使用简洁 Markdown 表格；表头名称与 HTML 版一致，不临时扩列；表格前后各空一行。非 A 股不使用 A 股资金流表，交易线索写入“价格与量能”段落。
- 风险提示使用短段落或编号，不使用小圆点堆叠；每条风险 1-2 句，语气克制。
- 正文重要事实和所有引用数据使用 `[1]`、`[2]` 这类来源编号；文末“信息来源”必须逐条对应，不允许所有事实或所有数据都挂同一个编号。当前价、涨跌幅、成交额、换手率、量比、资金流、板块、MA、MACD、RSI、PE TTM、一致预期等数据只要出现在报告里，就必须能回链到可打开来源。
- 全文使用中文标点：中文括号 `（ ）`、中文引号 `“ ”`、中文顿号和分号；证券代码、英文公司名、URL 除外。
- 语气为机构研究/交易台复盘：判断要明确，原因要说透，但不承诺收益、不使用指令式买卖建议。

### 5. 用户确认后生成 HTML

只有当用户明确表示“生成 HTML / 需要完整报告 / 做成页面 / 继续生成”时，才生成单文件 HTML：

- 生成 HTML 时必须先严格按照本 skill 的固定 HTML/CSS、数据契约、章节顺序、引用规则、价格区间带和 QA 清单生成自包含 HTML 文件。
- HTML 报告开头和对话版报告必须使用同一个生成时间：用户发起请求的 `report_request_time`。不得用文件写入时间、agent 完成时间或浏览器打开时间替代。
- 在交付 HTML 产物时，务必遵循下面的顺序：先使用 `FileBatchUpload` 工具拿到公网链接，作为附件；将附件带给 `app_builder_agent` 工具进行交付，要求**严格按照 HTML 源文件**进行渲染输出，不做任何代码变更。
- 文件自包含：CSS 内嵌，不依赖 CDN、外部字体、外部图片或外部 JS。当前版本不内嵌 TradingView Lightweight Charts、KLineCharts 或任何图表库，不生成 K线数据脚本或图表初始化脚本。
- 版式与本文后部“HTML 报告版式规范”完全一致。
- 首屏内部顺序必须固定：报告 header -> 公司一句话简介/开篇判断 `.lead` -> 摘要指标卡 `.metric-grid` -> 第一部分正文。不得插入 `.market-chart-module`、`Price Action / 价格走势`、`近期股价 K 线图`、`今日/1个月/3个月` tabs、图表容器或图表分隔线。
- 摘要指标卡只保留两行：标签 + 数值。不要出现“报告口径/今日反弹/元/人民币”等第三行重复说明；单位直接写进数值。
- 摘要指标卡下方必须有固定注释，说明 8 个行情摘要指标的时间口径，例如：`注：以上行情摘要截至 2026-06-12 13:30（交易日盘中）。` 或 `注：以上行情摘要为 2026-06-12 收盘口径。` 来源不写在注释中，而是通过每个数值的上角标回链到文末“信息来源”。
- 主章节顺序固定：
  1. 今日行情、量能与资金流向
  2. 板块联动与个股地位
  3. 技术面分析
  4. 消息面与隔夜观察
  5. 投行一致预期
  6. 综合评估
  7. 短期风险提示
- 不新增章节破坏固定版式；用户个性化问题必须嵌入 `.lead`、第一部分“价格与量能”、第三部分“技术面分析”、第四部分“消息面与隔夜观察”、第六部分“综合评估”和第七部分“短期风险提示”。

K线下线硬规则：

- 当前版本无论行情数据是否完整，都不生成 K线图相关 HTML、CSS、JS、数据或校验流程。
- 最终 HTML 中不得出现 `.market-chart-module`、`.chart-tabs`、`.tab-controls`、`.kline-panels`、`.chart-panel`、`Price Action / 价格走势`、`近期股价 K 线图`、`window.RECAP_CHART_DATA`、`LightweightCharts`、`KLineCharts`、`CandlestickSeries`、`HistogramSeries`、`data-range="today"`、`data-range="oneMonth"`、`data-range="threeMonth"`。
- 当前版本不拉取 30 个交易日日线；技术指标、压力位、支撑位和价格区间必须来自可核验数据源披露值、研报/公告/行情页，或带公式说明的自行计算。

第一轮默认输出：

- 直接在对话框输出“对话版完整日报”。

## 数据缺失处理

如果任何字段无法获取：

- 不要编造。
- 表格和指标卡缺失单元格统一写 `暂无`，避免长文本挤爆版式。
- 正文需要解释缺失时，可用短句说明“暂无可靠数据”。
- “信息来源”列出正文和表格实际引用且可打开核查的全部原始外部来源，包括公告、新闻、产业媒体、研报/预测，以及行情摘要、资金流、板块行情、技术指标、PE TTM、量比和一致预期等数据来源。缺失字段和被剔除字段不写入正文，也不写入来源列表。正文重要事实和所有展示数据必须用上角标引用回链到对应来源。不得把内部数据核验记录、本地 txt、本地材料包或中间文件作为来源。
- 若影响核心判断，降低评级确定性，并在风险提示中说明。

## 合规口径

本 skill 会生成研究讨论材料，不提供保证收益、确定性买卖结论或个性化交易建议。第六部分只保留“综合评估”和研究理由，不生成操作建议表格。

免责声明必须作为文末独立章节输出，紧跟“信息来源”。使用 `<section class="section source-note disclaimer-note">`，标题单独写 `免责声明`，下方固定三段正文；字体、字号、颜色、行间距、段间距与“信息来源”下方段落一致。固定文案见 [html-style.md](html-style.md) 的“免责声明”小节。

禁止：

- 承诺收益。
- 使用“必涨、稳赚、满仓”等绝对化表述。
- 引用无法验证的网络小作文作为事实依据。
