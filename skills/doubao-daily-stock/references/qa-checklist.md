## 生成前后复核清单

## 强制交付闭环

本清单只在用户明确确认生成 HTML 后作为交付门禁。用户未确认前，不得创建 HTML 文件；应先输出“对话版完整日报”，并在结尾询问是否生成完整 HTML 报告。

生成 HTML 后不得立即回复用户，必须按以下闭环执行到完全通过：

1. 第一轮全量复核：按“数据复核、内容复核、HTML 结构复核、信息来源复核”逐项检查，不得只抽查。
2. 发现任意失败项时，必须先修改 HTML/数据/脚本，再从“数据复核”重新开始全量复核。
3. 至少完成两轮复核：第一轮用于发现问题，第二轮用于确认修复后没有引入新问题。
4. 若第二轮仍有失败项，继续“修改 -> 全量复核”循环，直到所有硬规则通过。
5. 当前版本 K线模块已下线。最终回复用户前，必须确认页面不存在 K线图、图表标题、图表 tabs、图表容器、图表库、图表脚本、`window.RECAP_CHART_DATA` 或图表分隔线。

## 数据复核

生成 HTML 前逐项检查：

- 若用户提出个性化问题，必须已识别 `question_focus`：`target_price`、`short_trade`、`event_catalyst`、`position_advice` 或 `daily_recap`。
- 股票识别唯一，代码、交易所、币种正确。
- 必须已确定 `report_request_time`、`latest_valid_trading_day` 和 `data_as_of`。非交易日或停牌时，最近有效交易日必须向前推。
- 必须完成内部数据核验记录，且记录写明每个关键字段的原始外部来源链接、计算公式、核验结果和缺失原因。该记录不要求保存为 `.txt`，也不得作为“信息来源”。未完成核验记录不得验收。
- 每个关键数据必须能追溯到明确来源链接或带公式的自行计算：行情摘要、资金流、板块数据、MA/MACD/RSI、PE TTM、量比、新闻公告、研报/一致预期、风险事实。
- 所有重要行情、财务、资金、技术和新闻数据原则上用 2-3 个渠道核验。若来源冲突、日期不对、市场代码不对、单位不一致、复权口径不明或无法打开原文，应标记缺失并从报告中剔除。
- 当前版本不得获取或要求 1 分钟级行情、30 个交易日日线 OHLCV 或 90 个交易日行情。
- 不得用“缺少 30 日线”为由阻塞报告；技术指标、支撑压力和价格区间必须来自可核验数据源、研报/公告/行情页，或带公式说明的自行计算。
- 摘要指标 8 项齐全：当前价格、涨跌幅、日内区间、总市值、PE TTM、成交额、换手率、量比；字段缺失时统一写 `暂无`。
- A 股资金流数据来源明确，必须包含主力资金、北向资金和大单资金或明确缺失说明。非 A 股不适用 A 股资金流表，只需核验“价格与量能”段落中的交易结构线索。
- 板块标签不是泛泛概念，必须和股票走势最相关。
- MA、MACD、RSI、量比、PE TTM、资金流、板块数据和一致预期都有明确外部来源；若自行计算，必须有公式、输入来源和“自行计算”标记。
- 支撑位、压力位、震荡区间能由可核验数据源、研报/公告/行情页或带公式的自行计算解释。
- 新闻来源权威，消息面复盘日期必须在最近 3 日内；未来观察事项有实际日程或事件基础。
- 消息面新闻必须聚焦公司基本面、产业基本面或行业趋势；纯交易异动、资金流、大宗交易、龙虎榜不得重复作为第四部分新闻。
- 投行一致预期缺项时不编造，估算字段有公式。
- 若 `question_focus = target_price`，必须获取或估算压力位、支撑位、近 30 日高低点、投行一致预期或估值参照；不得凭感觉给目标价。
- 若 `question_focus = short_trade`，必须有成交额相对 5 日均值、可核验交易结构线索和关键波动区间，用于研究风险判断；不得生成具体买卖、仓位、止损或止盈指令。
- 若 `question_focus = event_catalyst`，必须核查用户提到的事件来源、日期、事实内容，并判断是否已被股价交易；若无法核实，必须明说。

## 内容复核

- 对话版完整日报必须已输出或在本轮明确给出核心结论；结构必须包含“核心结论”、一至七个主体章节、“信息来源”和“免责声明”。不得只输出 3-5 条摘要就进入 HTML。
- 未经用户确认，不得自动生成 HTML；若用户确认生成，HTML 内容必须与对话版日报观点一致。
- 若用户提出个性化问题，核心结论第一条、对话版日报第一部分、后续 HTML 的 `.lead`、第一部分和第六部分必须直接回答该问题。
- 第一部分已经合并为“今日行情、量能与资金流向”，不要再单独生成“今日行情总结”和“成交、量能与资金流向”两个重复章节。
- 第一部分必须把价格走势和成交量、换手率、量比、资金流结合分析，并解释造成价格和量能变化的基本面/行业/宏观/公告原因；没有明确原因时必须写明未见明确基本面催化，不得编造。
- A 股资金流表格必须保留；非 A 股不得生成资金流表格，应在“价格与量能”段落中说明成交、换手、相对均量、南向资金/期权/ETF 流向等可核验交易线索。
- 板块联动要说明个股相对板块强弱。
- 技术面不要只罗列指标，要给出趋势/动能/支撑压力结论。
- 第六部分必须只保留“综合评估”，不得生成操作建议表格，不得分空仓者、持仓者、短线参与者。
- 第六部分评级句后必须有 2-3 段评级说明，并包含基本面或行业趋势判断；不得只重复技术面、资金面或行情表现。
- 风险提示必须针对散户追涨杀跌、估值、业绩、资金、板块回撤等实际风险。
- 风险提示必须至少有一条直接对应用户问题，例如目标价失效、短线追高、事件兑现不及预期或利好已被定价。
- 不使用“必涨、稳赚、满仓”等绝对化语言。
- 不得以报告自身口吻输出直接投资操作建议，例如“建议买入 / 卖出 / 加仓 / 减仓”“适合买入”“应减仓”等。允许在客观转述、条件假设、风险提示或用户既有状态描述中出现相关词汇，但必须明确保持研究分析口径，不得形成操作指令或替用户作出交易决策。

## HTML 结构复核

用文本检查：

- 文件以 `<!DOCTYPE html>` 开头，以 `</html>` 结尾。
- 无 `<script src=`。
- 无外部 CSS、图片、字体。
- 无 TradingView Lightweight Charts、KLineCharts 或任何图表库内嵌脚本。
- 生成 HTML 时必须先严格按照本 skill 的固定 HTML/CSS、数据契约、版式规则和 QA 清单生成自包含 HTML 文件。在交付 HTML 产物时，务必遵循下面的顺序：先使用 `FileBatchUpload` 工具拿到公网链接，作为附件；将附件带给 `app_builder_agent` 工具进行交付，要求**严格按照 HTML 源文件**进行渲染输出，不做任何代码变更。
- 不得出现 `cdn.jsdelivr.net`、`unpkg.com`、`cdnjs.cloudflare.com` 等外部图表资源。
- 不得出现 `.market-chart-module`、`.chart-tabs`、`.tab-controls`、`.kline-panels`、`.chart-panel`、`Price Action / 价格走势`、`近期股价 K 线图`、`window.RECAP_CHART_DATA`、`LightweightCharts`、`KLineCharts`、`CandlestickSeries`、`HistogramSeries`、`data-range="today"`、`data-range="oneMonth"`、`data-range="threeMonth"`。
- 不得出现可见 tab 文案 `今日`、`1个月`、`3个月` 作为图表 tabs；这些词若出现在正文自然语言中不算失败，但不得以 `.tab-controls label`、`data-range`、radio input 或图表容器形式出现。
- 不得用运行时 JS 生成报告正文、章节、表格、信息来源或免责声明。当前版本原则上不需要任何 JS。
- 版式系统必须是单一最终版，不得出现 `REDESIGN OVERRIDES`、`晨报 v2/v3/v4/v5/v7`、`override patch`、`dashboard_template`、`历史样式`、`改版补丁` 等历史样式/迁移痕迹。
- 不得出现非本报告组件或可变布局 class：`.hero`、`.news-card`、`.timeline`、`.hero-evidence`、`.summary-card`、`.dashboard-card`、`.memo-card`。
- 不得使用 `<img>` 或图片失败后改变布局的脚本；报告新闻、公告和信息来源必须为文本结构。
- 不得新增 inline style；只允许价格区间带 `.zone-range`、`.zone-current`、`.zone-edge` 使用 `left` / `width` 动态定位。
- `@media` 不得把 `.two-col`、`.technical-dashboard`、`.report-header` 改回左右分栏，不得改变模块顺序。
- Header 中的 `meta-box` 只有一项：生成日期；必须放在公司业务/产品简介下方并左对齐，不得放到右侧自适应栏，不得包含报告类型、观点口径、证券代码，不得用表格或左右分栏。
- 首屏顺序必须是：`header.report-header` -> `.lead` -> `.metric-grid` -> 第一部分。`.lead` 与 `.metric-grid` 中间不得插入图表模块或图表分隔线。
- 章节编号依次为：一、今日行情、量能与资金流向；二、板块联动与个股地位；三、技术面分析；四、消息面与隔夜观察；五、投行一致预期；六、综合评估；七、短期风险提示。
- 摘要指标卡没有第三行说明；日内区间数值内含单位，如 `1126.90-1196.16元`；总市值数值内含币种，如 `约1.32万亿人民币`。
- 摘要指标卡颜色 class 正确：涨跌幅正数 `.positive`，负数 `.negative`，日内区间 `.neutral`，其他指标默认黑色。
- A 股资金流向表格表头必须是 `资金类别 | 流向趋势 | 复盘含义`，不得使用旧表头 `读数`。
- A 股资金流向表格必须使用 `<table class="fund-flow-table">` 和固定 `colgroup`：第 1 列 `.fund-col-category`、第 2 列 `.fund-col-trend` 等宽 `82px`，第 3 列 `.fund-col-meaning` 自适应并自动换行。不得给表格设置会强制窄屏横向拖动的固定 `min-width`。
- A 股资金流向表头三列居中且为黑色 `#2d2d2d`；正文 `资金类别` 居中，`流向趋势` 居中，`复盘含义` 左对齐；表头和正文均为 `11px`。
- 非 A 股不得出现 `.fund-flow-table`、`<h3>资金流向</h3>`、`主力资金`、`北向资金`、`大单资金` 三行 A 股专用资金口径。
- 投行一致预期有覆盖数据时必须使用 `<table class="consensus-table">`；表头和正文全部居中，包括所有数字列，表头和正文均为 `11px`。表头只能二选一：A 股/完整盈利预测使用 `年份 | 归母净利润 | 营业总收入 | 每股收益 | 市盈率`；港股/美股或券商评级口径使用 `机构名称 | 最新评级 | 目标价 | 日期`，展示最近约 5 家机构。
- 投行一致预期若经核验确无投行/券商覆盖、评级或目标价，不得生成表格；必须使用 `.consensus-empty` 纯文字说明暂无覆盖，并写清对买入该股票的机会和风险含义。不得出现全是 `暂无` / `未披露` 的表格，也不得把公司公告、IR 或资讯聚合页当作机构覆盖。
- 第六部分不得出现 `空仓者`、`持仓者`、`短线参与者`、`仓位 / 风控`、`止损`、`止盈` 等操作建议表格内容。
- 第七部分风险提示必须使用 `.risk-list` + `.risk-item`，不得使用浏览器默认 `<ul><li>` 小圆点或 `<ol><li>` 序号。
- 第七部分 `.risk-item` 的字体、字号、行高和段距必须与第六部分普通正文段落一致：`14px`、`line-height: 1.72`、段间距 `9px`。
- A 股第一部分“价格与量能”必须在上方，“资金流向”必须在下方；二者不得左右并排。非 A 股第一部分只保留“价格与量能”文本块，不生成独立资金流向块。
- 技术面“指标状态”必须在上方，“关键价格区间”必须在下方；不得左右并排。
- 第四部分“消息面复盘”必须在上方，“隔夜潜在资讯”必须在下方；不得左右并排。
- 技术面价格区间带图上只能出现三个数据文字：当前价、震荡区间低点、震荡区间高点；不得显示支撑位/压力位竖线或文字。
- `.zone-range.zone-oscillation` 必须是固定灰色覆盖条：`background: #d1d3d1`、`border: 1px solid #9aa0a6`；不得使用渐变、透明彩色、红绿压力/支撑色带或自定义颜色。
- `.zone-note` 必须是三项纵向排列：震荡区间、压力位、支撑位。
- 技术面 `指标状态` 必须使用 `.tech-indicator-table` 固定结构；必须有 `.tech-indicator-head` 表头和且仅有 3 个 `.tech-indicator-row`。三行顺序固定为 MA、MACD、RSI；每行四列固定为 `.tech-name`、`.tech-tag`、`.tech-analysis`、`.tech-judge`。
- 指标状态表表头 `指标 / 标签 / 分析 / 判断` 必须居中且为黑色 `#2d2d2d`；三行正文所有单元格字号必须与表头一致为 `11px`。正文列对齐固定为：`指标` 居中，`标签` 居中，`分析` 左对齐，`判断` 居中。
- 不得出现旧结构 `.tech-signal-grid`、`.tech-signal`、`.tech-copy`、`.tech-read`、`.tech-desc`、`.tech-badge`，也不得把判断独立成全宽横条。
- 文末必须先有 `<section class="section source-note" id="sources">` 的“信息来源”，紧接着有 `<section class="section source-note disclaimer-note">` 的“免责声明”。
- “免责声明”标题必须单独一行，不能写成段落开头的 `<strong>免责声明：</strong>`；正文必须为固定三段。

## 信息来源复核

报告最后“信息来源”必须写正文实际引用过、且可打开核查的来源。

写：

- 正文实际引用过的新闻来源。
- 正文实际引用过的公司公告、交易所/监管披露、公司 IR 来源。
- 正文实际引用过的产业媒体或行业研究来源。
- 正文实际引用过的研报/一致预期来源。
- 正文实际引用的原始外部来源；内部核验记录、本地 txt、本地材料包和中间文件不得作为来源。

不写：

- 没有进入正文、表格或评级判断的后台尝试来源。
- 缺失字段、冲突字段和被剔除字段来源。
- 来源列表下方的“行情数据截至……”或“本报告所有数据来自公开渠道”等说明段落。

来源与正文引用必须逐项核对：

- 正文每一个 `.cite-ref` 的 `href="#source-N"` 都必须存在对应 `id="source-N"`。
- 每个 `.source-row` 必须包含 `.source-index`、`.source-name`、`.source-date`、`.source-title`、`.source-link`。
- 每条来源行左侧为“编号 + 媒体/来源名称 + 日期 + 文章/公告/材料标题”，右侧为 `<a class="source-link" href="真实链接" target="_blank" rel="noopener">打开原文</a>`。
- `.source-link` 文案固定为 `打开原文`，并且必须位于每行最右侧；不得省略，不得改成纯文本，不得放在标题后方。
- 每个 `.source-link` 必须指向真实外部 URL、公告页面、媒体原文、公司/交易所/监管页面、产业/研报/预测原文或可打开数据页；禁止 `#`、空链接、`javascript:void(0)`、虚构 URL、搜索结果页、本地 txt、本地核验材料包或中间文件。
- 生成前必须实际验证链接可打开。若无法验证链接，就不得把该信息或数据写成事实；改为“暂未检索到权威来源”或从正文、表格、指标卡中删除。
- `.source-list` 不得有顶部灰色横线；“信息来源”标题下方只显示来源行，不得追加“说明：当前页面……”之类解释段落。
- 正式联网报告不得把公告、新闻、产业资料、一致预期等多个来源合并为单个 `[1]`；正文引用到几类可打开来源，文末就要有对应的多条 `.source-row`。只有完全基于单一外部原文时，才允许只有一条来源。
- 当前价、涨跌幅、日内区间、总市值、PE TTM、成交额、换手率、量比、资金流、板块行情、MA、MACD、RSI、压力/支撑、投行一致预期等展示数据必须有上角标和来源行；找不到来源的单元格必须写 `暂无`，不得写“暂未获取可信信息/暂未获取可信来源”等长句。
- 若某条信息只来自用户粘贴文本且没有外部链接，可在正文说明“用户提供材料显示……”，但不得在“信息来源”里生成本地材料包或假链接；正式 HTML 的信息来源只列可打开外部原文或数据页。
- 若某个重要事实无法找到来源，不得生成该事实；改写为“暂未检索到权威来源”或放入缺失说明。
- “消息面复盘”里的公司/行业新闻不得用资金异动短讯替代；必须能在“信息来源”里看到对应公告、权威财经媒体或产业媒体。

## 免责声明复核

“免责声明”必须紧跟“信息来源”之后，固定三段：
